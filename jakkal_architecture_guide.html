01 · System Architecture

How JAKKAL is built
under the hood.

JAKKAL follows a layered mobile-first architecture. Each layer is independent — you can upgrade or swap one layer without rebuilding everything. This is critical for a startup because it lets you ship fast and improve incrementally.

Layer 1 — Client / Frontend
What users see and touch
React Native (iOS + Android) JAKKAL Lite (USSD/SMS) Progressive Web App (PWA) Voice UI / Icon-Only mode
↕
Layer 2 — API Gateway
Single entry point for all requests
REST API WebSocket (live traffic) Auth / Rate limiting HTTPS only
↕
Layer 3 — Core Services (Backend)
Business logic — each service is independent
🎙 Wolof ASR Service 🗺 Traffic Engine 🚌 Bus Tracker 🚕 Ride-Hail Integrations 🛡 SafeRoute Service 👤 User / Auth Service 📱 USSD Gateway 🔔 Push Notifications
↕
Layer 4 — Data Layer
Storage, caching, real-time streams
PostgreSQL (users, routes) Redis (live traffic cache) TimescaleDB (traffic history) S3 / Object Storage (audio) Message Queue (Kafka/SQS)
↕
Layer 5 — Infrastructure & AI
Cloud, AI models, third-party integrations
AWS / GCP (cloud) Google Maps Platform Wolof ASR Model (WAXAL) Yango / Heetch APIs Wave / Orange Money Africa's Talking (USSD) Firebase (push notifs)

Key architectural principle: Build the Wolof Voice AI and Traffic Engine as independent microservices from day one. This lets you upgrade the AI model without touching the rest of the app, and sell/license these services to other companies later as standalone products.

03 · Build Options

From scratch vs.
AI-assisted building.

There are five realistic paths to building JAKKAL. Each has a different cost, speed, and technical requirement. The right choice depends on your team's skills, your runway, and how fast you need to move.

Option A

Build from scratch — Custom code

Hardest · Most control

Your technical co-founder writes every line of code using standard engineering tools: React Native for mobile, Node.js/Python for backend, PostgreSQL for data. Every feature is custom-built to your exact specifications.

Advantages
✓ Full ownership of every line of code
✓ No vendor lock-in or platform risk
✓ Can optimize for Dakar-specific needs
✓ Best long-term scalability
✓ IP is 100% yours to license or sell
✓ No per-user or per-feature fees at scale
Disadvantages
✗ 12–18 months to MVP minimum
✗ Needs 2–3 senior engineers
✗ Highest upfront cost ($80K–$150K)
✗ No safety net if engineers leave
✗ Everything is your responsibility to debug

07 · Data Requirements

What data JAKKAL needs.

JAKKAL is a data-intensive app. Knowing what data you need, where it comes from, and how to handle it legally is critical — especially given Senegal's GDPR-equivalent data protection law (Loi n° 2008-12).

Traffic Data

Real-time & Historical Traffic

Source: Google Maps Traffic API (real-time), community crowdsourcing (later)
What you collect: Road speed, congestion levels, incident reports, flood/road closure alerts
Frequency: Every 30 seconds for live map; hourly snapshots for history
Storage: TimescaleDB (time-series optimized) — keep 2 years of history for AI model training
Privacy: No personal data — aggregate road data only
Transport Data

Bus & Vehicle Data

Sources: CETUD API (BRT official), Dem Dikk API, community GPS (informal buses)
What you collect: Bus GPS positions, route IDs, arrival times, capacity status
Frequency: Every 15 seconds per vehicle
Storage: Redis (live positions), PostgreSQL (routes/schedules)
Partnership needed: CETUD formal data sharing agreement — start this now
User Data

User & Behavioral Data

What you collect: Phone number, home/work locations, saved routes, trip history (anonymized), language preference
What you NEVER collect: Full real-time GPS tracks (collect only start/end), contacts, financial data beyond payment confirmation
Retention: Active data: 12 months. Anonymized aggregate: forever for AI training
Legal: Explicit consent required. Must register with CDP (Commission des Données Personnelles, Senegal)
Voice Data

Wolof Audio Data

MVP approach: Use pre-built Wolof ASR model (WAXAL open-source dataset)
Training data: Build your own Dakar-specific voice dataset over time — place names, slang, neighborhood names
Privacy: Do NOT store voice recordings without explicit consent. Process → discard. Only store anonymized intent data
Volume needed: 1,000+ hours of Wolof audio to meaningfully improve the model
Mapping Data

Geographic & POI Data

Sources: Google Maps API, OpenStreetMap (free), custom Dakar POIs
What to build custom: Informal bus stop locations (not in Google Maps), market day calendars, neighborhood boundaries, flood-prone zones
SafeRoute data (Phase 2): Street lighting data, police station locations, crowd density — requires field research
External APIs

Third-Party Integrations

Yango API: Request access via their partner program. Provides ride availability, pricing, booking
Heetch API: Similar — contact their Africa partnerships team
Wave / Orange Money: For in-app payments — register as a merchant
Africa's Talking: USSD + SMS gateway. Has a Senegal presence. Developer-friendly API
Firebase FCM: Free push notifications up to 1M/day
08 · Security

Keeping JAKKAL safe.

Security is not optional — especially when you handle location data for millions of users. A single data breach can kill a startup instantly. Build security in from day one, not as an afterthought.

🔐

Authentication & Access

  • • Phone number + OTP (no passwords — easier for users)
  • • JWT tokens with short expiry (1 hour access, 30 day refresh)
  • • Rate limiting on all API endpoints (prevent abuse)
  • • Never store raw OTP codes — hash immediately
  • • 2FA for admin/operator dashboards
🌐

Data in Transit

  • • HTTPS everywhere — no HTTP at all
  • • Certificate pinning in the mobile app
  • • TLS 1.3 minimum
  • • WebSocket connections also over WSS (secure)
  • • VPN for internal service communication
🗄️

Data at Rest

  • • Encrypt sensitive fields in database (phone numbers, location history)
  • • AWS KMS or GCP KMS for key management
  • • Automated daily database backups, encrypted
  • • Separate staging and production databases
  • • Never put real user data in development environments
📋

Legal Compliance (Senegal)

  • • Register with CDP (Commission des Données Personnelles)
  • • Privacy policy in Wolof AND French
  • • Explicit opt-in consent for location data
  • • Right to deletion — user can delete account + all data
  • • Data must be stored in Senegal or ECOWAS region where possible
🛡️

App Security

  • • Code obfuscation on Android APK
  • • No hardcoded API keys in mobile app code
  • • Runtime Application Self-Protection (RASP)
  • • Regular dependency audits (npm audit, snyk)
  • • Google Play Protect integration
🔍

Monitoring & Incident Response

  • • Sentry for error tracking (free tier available)
  • • Datadog or CloudWatch for infrastructure alerts
  • • Set up alerts for unusual API call volumes (bot detection)
  • • Have an incident response plan before launch
  • • 72-hour breach notification rule (under Senegal law)
09 · Performance & Reliability

JAKKAL must work
in Dakar conditions.

Dakar has highly variable internet connectivity. Some users are on 4G LTE; many are on 2G/3G or intermittent WiFi. Your app must work under these conditions — otherwise you're excluding the people you're trying to serve.

📶

Low-connectivity Design

  • • Offline mode: Cache last-known traffic + route. App still works without internet for known routes
  • • Aggressive compression: All API responses gzipped. Images WebP format only
  • • Lazy loading: Only load map tiles for the visible area
  • • App size: Target under 20MB APK download (many users on limited data plans)
  • • Progressive enhancement: Basic route info loads first; voice AI loads if connection allows
⚡

Scalability Plan

10K–100K users: Auto-scaling groups, read replicas, CDN for static assets. ~$500–1,500/mo
100K–1M users: Microservices separation, database sharding, multi-region. ~$3,000–8,000/mo
Key bottleneck: The Wolof ASR service is CPU-intensive. Dedicate a GPU instance for it early. Cost ~$200–400/mo.
🔁

Reliability & Uptime

  • • Target: 99.5% uptime (allows ~44 hours downtime/year)
  • • Multi-availability-zone deployment on AWS
  • • Database replication + automatic failover
  • • Health checks every 30 seconds
  • • Graceful degradation: if ASR is down, fall back to French text input
  • • Status page (statuspage.io) — be transparent with users
